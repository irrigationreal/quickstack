# QuickStack

QuickStack is the agent wrapper for the `quickstack` CLI. Treat the CLI as the product surface and this skill as guidance: call `quickstack`, relay its structured questions/errors, and avoid reimplementing deployment logic in prose. The job is to turn local project intent into QuickStack apps, builds, secrets, ports, domains, and deployments while hiding Kubernetes unless the user explicitly brings Kubernetes manifests.

## Install

Install the server-matched CLI from the target QuickStack server:

```sh
curl -sSL https://<server>/api/cli/install | sh
```

The script installs `quickstack` to `~/.quickstack/bin/quickstack` and writes the server URL to `~/.quickstack/config.json`. If `~/.quickstack/bin` is not on `PATH`, tell the user to add it or invoke the binary with the full path.

## Core rules

- Never store API keys, registry passwords, `.env` values, kubeconfig, upload tokens, or secret env values in `.quickstack` project state.
- Do not raw-apply Docker Compose or Kubernetes YAML. Convert common safe shapes into QuickStack app specs. Ask or refuse for unsafe primitives.
- Ask before making a topology decision that changes what will be public, stateful, or shared.
- For a new app, ask whether to use an autogenerated QuickStack hostname or a custom hostname. If `.quickstack` already links the folder to an app/domain, reuse that unless the user asks to change it.
- Prefer one clear deployable service. If the repo contains multiple public services, present the plan and ask.
- In CI or when `QUICKSTACK_NONINTERACTIVE=1`, do not ask interactively. Fail with a structured list of questions that need answers.
- Do not deploy a managed upload until the app points at the produced managed image. If the server cannot reserve/build/update safely yet, stop with a capability error instead of deploying a placeholder or stale image.
- Use `quickstack ...` for launch, deploy, config, packaging, and secrets workflows. Do not call QuickStack HTTP APIs directly from the skill instructions unless debugging the CLI itself.

## Required credentials

The preferred local credential store is `~/.quickstack/config.json`:

```json
{
  "url": "https://quickstack.example.com",
  "apiKey": "qstk_..."
}
```

Use `quickstack setup --url <url> --api-key <qstk_key>` to create that file with `0600` permissions. If the user gives you a `qstk_` API key during setup or says to configure QuickStack deploys, immediately save it with `quickstack setup` or the agent's sensitive config before running deploy commands. Do not ask them to paste it again unless the save failed. `QUICKSTACK_URL` and `QUICKSTACK_API_KEY` remain valid environment overrides for CI and temporary sessions.

Never read credentials from project state and never write them there. `.quickstack` project state is only for secret-free deployment state such as app id, project id, hostname, service root, mode, port, build hash, and image reference.

## Workflow

1. Invoke the CLI first.
   - Prefer `quickstack launch "$PWD" --json` for first-run deploys.
   - Prefer `quickstack deploy "$PWD" --json` for repeat deploys with `.quickstack` state.
   - If the CLI returns `outcome: "question"` or a structured ambiguity error, ask the user using the CLI's question/options. Do not invent a separate decision tree.
   - Use lower-level `detect`, `package`, or `api` only when developing or debugging the CLI.

2. Ask only when the answer changes deployment.
   Ask for project selection, service selection, port, public domain choice for a new app, one-app-vs-two-app topology, Compose/Kubernetes mapping, database/cache treatment, secret/public env classification, build command and output folder, branch/environment target, private registry credentials, or whether to reuse an existing app.

3. Let the CLI call QuickStack APIs.
   - For normal operation, run the `quickstack` CLI and relay its output.
   - For existing image mode, the CLI can ensure and deploy using `--image`.
   - For managed uploads, the CLI must only upload when the server supports the full reserve/build/update/deploy sequence. If a new app needs an id first and the server lacks that capability, relay the CLI's capability error. Never deploy after bootstrap ensure until a produced image is set.
   - For Compose/Kubernetes imports, use the CLI import/launch flow. If the backend normalized import endpoint is unavailable, stop and print the import plan rather than raw-applying manifests.

4. Poll and report.
   - Poll deploy status and HTTP readiness when endpoints exist.
   - Print phase logs: detected, planned, packaged, uploaded, built, ensured app, deployed, ready.
   - Write secret-free state under `.quickstack/` after successful config/build/deploy state changes.

## Secret handling

- Treat `.env`, Compose `env_file`, Compose `secrets`, and Kubernetes `Secret` values as secret by default.
- Only classify env values as public when they look intentionally public or the user says so.
- Redact secrets in summaries. Show names, not values.
- Registry credentials are secret inputs and should feed QuickStack Docker pull-secret support. Do not put them in project state.

## `.quickstack` state

Use a directory layout for multi-service safety:

```json
.quickstack/index.json
.quickstack/apps/<appId>.json
```

State may include project id, app id, service root, branch/environment label, mode, port, domain, build command/output, last content hash, last build id, and last image reference. State must not include secret values.

If saved branch/environment differs from the current branch/environment, ask whether to reuse the app, create/update a preview app, or change mapping.
